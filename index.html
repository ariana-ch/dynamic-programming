Chapter 1

What dynamic programming actually is

Start with the right mental model before memorising any recurrence.

Foundation
Definition. Dynamic programming is a method for solving a problem by breaking it into overlapping subproblems, solving each subproblem once, and reusing those answers.

The two phrases that matter are overlapping subproblems and reusable state. If solving one smaller piece gives information you will need again, you are in dynamic-programming territory.

Intuition. DP is not “use a table.” A table is only the storage. The real idea is to choose a state that contains exactly the information needed to continue optimally or count correctly.
Example. In Fibonacci numbers, the value of F(n) depends on F(n-1) and F(n-2). If you recurse naively, you recompute the same quantities many times. DP avoids that waste.
Do not call every recursive solution “dynamic programming.” If you recurse without caching, and the same subproblem appears repeatedly, you have not yet done the dynamic-programming part.

Chapter 2

The DP design loop

Learn a repeatable process for constructing DP solutions.

Core workflow
The DP design loop.

Step 1. Define the state.
Step 2. Define the meaning of the state in one sentence.
Step 3. Write the recurrence or transition.
Step 4. Set base cases.
Step 5. Choose evaluation order or memoization strategy.
Step 6. Identify the final answer state.
Step 7. Check complexity and possible compression.
State. A state is the minimal collection of variables needed to describe a subproblem completely.
Example. For climbing stairs with steps of size 1 or 2, the state can be dp[i] = number of ways to reach step i. Then dp[i] = dp[i-1] + dp[i-2].
Many DP failures come from a vague state meaning. If you cannot finish the sentence “dp[i][j] means ...”, your recurrence is probably not trustworthy yet.

Exercises

Exercise 2.1

For minimum cost to reach index i in an array, what could dp[i] mean?

Show model answer
dp[i] can mean the minimum total cost needed to arrive at index i.
Chapter 3

Memoization and tabulation

Understand the two main implementation styles and when to choose each.

Implementation mindset
Memoization. Top-down recursion plus caching.
Tabulation. Bottom-up iteration in an order that guarantees dependencies are already known.
Memoized Fibonacci.
memo = {}

def fib(n):
    if n <= 1:
        return n
    if n in memo:
        return memo[n]
    memo[n] = fib(n-1) + fib(n-2)
    return memo[n]
Tabulated Fibonacci.
def fib(n):
    if n <= 1:
        return n
    dp = [0] * (n + 1)
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i-1] + dp[i-2]
    return dp[n]
Rule of thumb. Memoization is often easier when the state graph is irregular. Tabulation is often easier to optimise and debug once the order is clear.

Chapter 4

One-dimensional DP and decision chains

Master the simplest but most common DP shape.

Linear states

In one-dimensional DP, the state is indexed by one main progress variable such as position, value, amount, or length.

Typical meanings. dp[i] may mean a minimum cost, maximum score, boolean feasibility, or number of ways.
Coin change counting. If dp[s] is the number of ways to form sum s, then the transition depends on whether order matters. That design decision changes the loop order completely.
For combinations, iterate over coin types outside and sums inside.
For permutations, iterate over sums outside and coin types inside.
Equal. Some HackerRank problems in the DP area are best solved by noticing a small one-dimensional cost function after reformulation rather than using a large table. Equal is a good example of this style. ([hackerrank.com](https://www.hackerrank.com/challenges/equal/forum/comments/65311?utm_source=chatgpt.com))
A one-dimensional DP can still be wrong if the state is missing hidden information such as whether previous elements were chosen, or if the loop order accidentally reuses updated values too early.

Exercises

Exercise 4.1

In a staircase problem where order matters, should the outer loop usually be over target sum or over move type?

Show model answer
Target sum. That way each state aggregates all ways to arrive there in order-sensitive fashion.
Chapter 5

Two-dimensional DP and sequence alignment

Handle problems where progress in two objects must be tracked simultaneously.

Matrix DP
Canonical state. dp[i][j] often means the answer for prefixes A[:i] and B[:j].
LCS pattern. In longest common subsequence, if the last elements match, extend diagonally. Otherwise take the better of skipping one side. The HackerRank DP domain includes The Longest Common Subsequence, which is a classic example of this template. ([hackerrank.com](https://www.hackerrank.com/challenges/dynamic-programming-classics-the-longest-common-subsequence/forum/comments/35546?utm_source=chatgpt.com))
if A[i-1] == B[j-1]:
    dp[i][j] = dp[i-1][j-1] + 1
else:
    dp[i][j] = max(dp[i-1][j], dp[i][j-1])
Interpretation trick. Prefix-based states are powerful because deleting the last character cleanly reduces the problem size.
Variants. Edit distance, longest common substring, shortest common supersequence, and alignment-scoring problems all live near this template.

Exercises

Exercise 5.1

In prefix DP for LCS, what does dp[0][j] equal?

Show model answer
Zero, because one sequence is empty.
Chapter 6

Counting DP and combinatorial transitions

Learn to count huge families of objects without double-counting or missing cases.

Counting

Many hard DP problems are counting problems modulo a prime. In these questions, correctness often depends more on the state meaning and loop order than on algebra.

Typical state meanings. Number of ways to form a sum, number of paths to a cell, number of sequences of length i with some property, or number of ways after processing the first i items.
Grid Walking. A common hard pattern is to solve one dimension first, then combine dimensions with another DP plus binomial coefficients. Grid Walking is a representative example in HackerRank’s dynamic-programming area. ([hackerrank.com](https://www.hackerrank.com/challenges/grid-walking/forum/comments/60580?utm_source=chatgpt.com))
Design lesson. When the full state seems too large, ask whether one coordinate or one component can be solved independently and then composed.
Counting DP often fails due to overcounting. Always ask whether two different transition paths represent genuinely different objects or the same object described twice.

Exercises

Exercise 6.1

Show model answer

Chapter 7

Subset, bitmask, and profile states

Handle problems where the state must remember a set of acquired items or visited choices.

State compression
Bitmask DP. Use a binary integer to represent a subset, where bit b indicates whether item b is already included.
Canonical state. dp[mask][i] can mean the best value after selecting the subset mask and ending at item i.
Hybrid graph-state pattern. Some problems are solved by shortest paths over states like (node, mask), where mask records acquired types. HackerRank has examples where the right solution is effectively a graph search over such compressed states rather than a plain rectangular table. Discussion around Synchronous Shopping highlights exactly this pattern. ([hackerrank.com](https://www.hackerrank.com/challenges/synchronous-shopping/forum/comments/521824?utm_source=chatgpt.com))
When to think bitmask. The moment the problem says “collect a subset of k things,” “visit selected items,” or “remember which features are already satisfied,” you should test whether 2^k is small enough.

Exercises

Exercise 7.1

If k = 10 item types, how many masks are there?

Chapter 8

Interval, partition, and tree DP

Learn the high-value patterns that appear once simple prefix states are no longer enough.

Structural DP
Interval DP. States describe subarrays or substrings such as dp[l][r].
Partition DP. States describe the best value after splitting the first i items into some number of groups.
Tree DP. States describe answers for subtrees, often with one or more conditions on the parent-child relation.
Interval example. If combining interval [l, r] depends on trying every split point k, then the transition often looks like dp[l][r] = min/max over k of dp[l][k] + dp[k+1][r] + merge_cost.
Tree example. Define a subtree state, write how child information combines, and be explicit about whether the parent is selected, constrained, or absent.
Structural DP usually fails when the state ignores one small but crucial condition, such as whether the boundary has already been matched, whether a parent is chosen, or whether a split count is fixed.

Exercise 8.1

Chapter 9

State-space shortest paths and hybrid DP

Recognise when the right state exists but should be explored with Dijkstra, BFS, or DAG relaxation rather than a plain table scan.

DP + graph view

Some problems have a perfectly valid DP state, but the transition graph is sparse or irregular. Then the best implementation is often a graph search over states rather than nested loops.

State-graph view. Treat each DP state as a node and each valid transition as an edge.
Examples. Shortest path with extra resources, node plus bitmask states, and “best answer reaching this configuration” problems often fit this pattern. The discussion of Synchronous Shopping uses a lowestTravelTime[node][mask] state exactly in this spirit. ([hackerrank.com](https://www.hackerrank.com/challenges/synchronous-shopping/forum/comments/521824?utm_source=chatgpt.com))
Decision test. If your DP table feels right but the natural evaluation order is unclear, ask whether the states form a graph that should be explored by a shortest-path algorithm.
Chapter 10

Recognition checklist and debugging

Build the instincts that let you spot a DP problem and repair a broken solution.

Pattern recognition
Recognition checklist.
Debugging checklist. State meaning, base cases, transition correctness, loop order, modulo placement, impossible states, and memory compression safety.
If the answer is close but wrong, suspect one of three things first: an off-by-one in prefixes, double-counting due to loop order, or a state that forgot one necessary condition.

Module 3

Optimisation techniques

Prepare for the difference between a correct recurrence and a solution that actually passes.

Advanced practice
Space compression. If a state depends only on the previous row, previous layer, or nearby values, reduce memory by keeping only what is needed.
Transition pruning. Reduce candidate predecessors by monotonicity, sorting, or problem-specific structure.
Precomputation. Prefix sums, combination tables, bit counts, and valid-mask lists can turn a slow transition into a feasible one.
Mindset. Many hard DP problems are not conceptually harder than medium ones. They simply require better state compression, precomputed helpers, or a more careful order of evaluation.
